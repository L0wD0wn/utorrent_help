µTorrent User Manual

Formatting Conventions

Formatting conventions for anyone who updates the manual... Yeah, who cares, right? Well, I'm a consistency freak, and really would prefer it if everything were kept (*gasp*) consistent. So what do we do about it? We set some guidelines about how formatting, terminology, and stuff in general are done!

TERMINOLOGY

Yes, I use specific terminology for stuff in an attempt to keep word confusion to a minimum for the reader. Here are some of the words I can think of:

  • .torrent file: used when referring to .torrent files
  • torrent job(s): refers to any of the torrents listed in the torrent jobs list
  • torrent contents: the files being downloaded
  • torrent: used only if whatever you're trying to describe fits in none of the above, and there is no other reasonable way to refer to it
  • KiB/s, MiB/s, etc: binary prefixes are used rather than SI prefixes

If there is a word that's worth defining for the user, make sure you add it to the glossary.

BASIC FORMATTING

Here are some of the more obvious formatting conventions I chose to use:

  • <strong></strong> rather than <b></b>
  • <em></em> rather than <i></i>
  • key term for key terms (vocabulary)
  • "interface text" for text that comes directly from the interface (when yo're referring to the exact name of an interface control, for example). Do note the quotations around the <em></em> tags. This only for when you are mentioning the control, not for when you are listing it for description.
  • modifier+key for keyboard combinations
  • mouse action for mouse actions (like "double-clicking," or "secondary-click")
  • value for text that refers to some specific value (like "30 KiB/s," or "MaxConnections in the Registry")

ANCHORING

Anchoring... I actually use something along the lines of

<a id="SOME_NAME"></a>

instead of

<a name="SOME_NAME"></a>

since the latter was deprecated in XHTML 1.1. If something you're adding an anchor to is a part of another heading, you should use

<a id="HEADING_ANCHOR_NAME.SUB_ITEM"></a>

where HEADING_ANCHOR_NAME is the anchor name for the heading that the item falls under. Yes, there is sub-sub-anchoring too, if you have a heading with a sub-heading with something that needs anchoring. You can check most of the AppendixA pages for examples if I just confused you.

What should get an anchor? ANY control getting described, clearly. Important sections of other stuff though as well. Taking a look at the source for all the files would give a good sense of what I felt needed to be anchored, though. Why do anchoring at all? It makes it easier to link to later.

The above examples are actually simpler than they really are or should be. The full anchoring I use is self-referencing, like

<a class="name" id="CONTROL_NAME" href="#CONTROL_NAME">CONTROL NAME</a>

(example: CONTROL NAME)

Yes, I keep the <a></a> inside of other tags (and you should too) because other tags might have some weird margins, which would cause the anchored linking to have some space above. If you're not sure what I'm referring to... well, just do it this way please? :)

Why should anyone bother to do that? Because it makes it easier to actually get a link to the particular item. Say, for example, I wanted to get the link to a particular item in the FAQ. Since it has no TOC, I would have to look through the source to find the ID to append onto the URL. BUT, if you did it my way, the user could simply click the link and copy the address bar, or right-click the anchor and copy the location. Ease of access!

When creating anchor IDs, there are some characters that should be removed and/or changed:

  • µ (&micro;) should be changed to u
  • most non-alphanumeric characters, like commas, single- or double-quotes, semicolons, or questionmarks should be removed (well, they're not valid XHTML anyhow)
  • anything within parentheses should be excluded from the anchor ID, so avoid using parentheses for links

LISTING STUFF

Meh, I rarely use ordered lists (<ol></ol>), and prefer to explain everything in paragraphs or <ul></ul> instead... Do realize there are "uienum" and "sublist" classes for lists, so make use of them properly (please? xD).

If a list item has a specific theme with a name (like when you're listing interface controls), put a <strong></strong> around the name/title of the list item... Since that description was probably more confusing than helpful, just take a look at the other lists if you're in doubt about what to do (yes, I said that before, didn't I? :P)

MISCELLANEOUS

The "micro" sign should be written as &micro; and not as an actual "micro," as everything is encoded in ANSI.
